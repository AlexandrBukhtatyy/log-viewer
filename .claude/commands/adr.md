description: Создать новый Architecture Decision Record в docs/adr/
---

Создай новый ADR на основе текущего разговора и аргументов: $ARGUMENTS

Шаги:

1. Прочитай `docs/adr/0000-template.md`.
2. Найди наибольший существующий префикс `NNNN-` среди файлов `docs/adr/NNNN-*.md` (исключая `0000-template.md`). Следующий номер = `max + 1`, с ведущими нулями (4 цифры). Если ADR ещё нет — следующий номер `0001`.
3. Сгенерируй kebab-case slug из сути решения (3-6 слов, английский).
4. Создай `docs/adr/NNNN-<slug>.md` по шаблону, заполнив:
   - **Title** — суть решения, человеческим языком.
   - **Status** — `proposed` (если пользователь явно не сказал «accepted»).
   - **Date** — сегодняшняя дата, формат `YYYY-MM-DD` (`date +%Y-%m-%d`).
   - **Context and Problem Statement** — что обсуждалось до этого момента и почему встал вопрос. Кратко, без воды.
   - **Considered Options** — альтернативы из разговора. Если их явно не было — добавь хотя бы «do nothing» и одну реалистичную альтернативу.
   - **Decision Outcome** — выбранный вариант + обоснование (1-3 предложения).
   - **Consequences** — Good / Bad / Neutral последствия.
   - **Diagram** — опционально, но **рекомендуется** для архитектурных ADR. Только Mermaid, тип уместный (`flowchart` / `sequenceDiagram` / `erDiagram` / `stateDiagram-v2`). Если диаграмма не нужна — удали секцию целиком.
   - **Links** — если в разговоре упоминались PR/issue/смежные ADR/внешние материалы.
5. Допиши новую запись в секцию `## Index` файла `docs/adr/README.md` в формате:

   ```
   - [NNNN. <Title>](NNNN-<slug>.md) — proposed
   ```

   Запись добавляется **в конец** списка (порядок по номеру).
6. Сообщи путь к созданному файлу markdown-ссылкой `[NNNN-<slug>.md](docs/adr/NNNN-<slug>.md)`.

Важно:

- Никаких диаграмм кроме Mermaid.
- Не редактируй другие файлы.
- Не запускай тесты, билд или линт.
- Не делай git-коммит.
