# Процесс формирования и поддержки Roadmap

## Context

[docs/ROADMAP.md](../ROADMAP.md) появился как stage 0 — статичный список ближайших планов. Чтобы он не превратился в «свалку идей, которую забыли обновить», нужен лёгкий процесс: как идея попадает в roadmap, как двигается по этапам, как уходит из него, и кто это всё поддерживает.

Цель: сформулировать минимальный, выполнимый ритм работы с roadmap'ом — чтобы он отвечал на три вопроса: «что мы делаем сейчас», «что следующее», «что мы решили не делать».

## Scope (черновик — обсудить)

Решения, которые надо принять до фиксации процесса:

1. **Источник истины.** Где живёт «канонический» список идей: `docs/ROADMAP.md` (плоский ручной список), `docs/plans/*.md` (по файлу на идею), `docs/backlog/*.md` (если внедрим backlog-workflow), GitHub Issues, GitHub Projects. Сейчас фактически — смесь plans + Issues, нужно выбрать ведущий.
2. **Этапы.** Простой lifecycle: `idea → planned → in-progress → done | dropped` (как обсуждали в feature-backlog-workflow). Или сложнее — с приоритетами, оценками, milestone'ами.
3. **Кто двигает.** Один автор (Alexandr) ведёт roadmap руками или любой контрибьютор может предложить PR с правкой `ROADMAP.md` / нового плана.
4. **Ритм синхронизации.** Раз в спринт / раз в месяц / continuously: автор проходится по списку, чистит «зависшие» элементы, перемещает done в «Shipped» секцию.
5. **Отметка выпуска.** В ROADMAP — отдельная секция `## Shipped` (по версиям / по дате)? Или удалять из roadmap и доверять `git log` + ADR-индексу?
6. **Связь с ADR.** Какие пункты roadmap дают начало ADR? Правило: «всё, что меняет архитектурный контракт». Связь односторонняя: план может породить ADR, наоборот — нет.

## Подход (предварительный)

Минимальная конструкция, которую можно потом расширить:

1. **Идея** записывается одним из двух способов:
   - Маленькая → GitHub Issue с label `idea`.
   - Большая (≥ день работы или меняет UI) → `docs/plans/<slug>.md` со статусом-заголовком и разделами Context/Design/Critical files/Verification.

2. **Roadmap-страница** ([docs/ROADMAP.md](../ROADMAP.md)) — секционирована по lifecycle:

   ```
   ## In progress
   ## Planned (next ~quarter)
   ## On the horizon
   ## Shipped (last 90 days)
   ## Dropped
   ```

   Каждый пункт — ссылка на `plan` или Issue. Roadmap **не** дублирует содержание плана, только заголовок + краткий one-liner + ссылка.

3. **Движение пункта**: автор делает PR, переносящий строку из одной секции в другую. PR-описание объясняет почему. Это создаёт версионную историю в git.

4. **Регламент проверки**: раз в месяц автор проходит ROADMAP сверху вниз и:
   - Pull'ит `Planned` в `In progress` если начал.
   - Pull'ит `In progress` в `Shipped` если ADR/PR закрыты.
   - Двигает в `Dropped` всё, что не двигалось 90+ дней и не имеет PR.

5. **PR template** для roadmap-правок (опционально, отдельный plan для PR-инфраструктуры): чек-лист «ссылка на план/issue, причина движения, обновлена дата».

## Critical files (предварительно)

Новые:

- `.github/PULL_REQUEST_TEMPLATE/roadmap.md` (опционально) — чек-лист для roadmap-правок.

Изменяются:

- [docs/ROADMAP.md](../ROADMAP.md) — переразметить под пять секций (`In progress / Planned / On the horizon / Shipped / Dropped`).
- [CONTRIBUTING.md](../../CONTRIBUTING.md) — секция «Roadmap process» с описанием движения пунктов.
- [docs/adr/](../adr/) — возможно новый ADR `NNNN-roadmap-process.md` после принятия решения (это процесс, тянет на architectural decision).

## Verification

1. **Smoke**: пройти готовый план `column-click-sort.md` через новый процесс — добавить его в секцию `Planned`, потом перенести в `In progress` (когда начнём), потом в `Shipped` (после merge).
2. **Read-pass**: открыть ROADMAP.md глазами «нового человека» — за 30 секунд понятно, что делаем сейчас, что следующее, что отвергли.
3. **Stale check**: ни одного пункта в `In progress` без PR/branch'а старше месяца.
4. **CONTRIBUTING.md** ссылается на новый процесс и объясняет, как добавить идею.
